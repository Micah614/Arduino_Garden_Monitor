Introduction: Arduino Prototyping and the IoT

Internet-connected devices are currently found in cars, homes, tablets, watches, cell phones, surveillance systems, strap-on monitors, virtual reality equipment, and more. Connected device commerce is expected to expand to a $581.85 billion industry by 2032, according to a recent report by precedenceresearch.com. The increasing prevelance of network-driven devices in the past decade has driven researchers to adopt the term “internet of things”, or “IOT”, for short. This project will explore the fundamental characteristics of IOT devices by developing a Wi-Fi enableed greenhouse monitoring station using Arduino single-board computer chips, the C++ programming language, as well as HTML, CSS, and little bit of JavaScript. The first section of this website is dedicated to completing a series of FreeNove Arduino Project Board tutorials to learn about the Arduino compatible hardware and the integrated development environment (IDE), with the assistance of a Freenove® project board tutorial.

Figure 1.) The project board we will be using to learn about driver development, Arduino prototyping, and the IDE environment in weeks 2-5 of the semester.

Project Vision Statement

The goal of this project is to develop a wide-area network transceiver device using Arduino compatible boards, hardware, and sensors to relay data about the monitored state colleced by the greenhouse sensors. This project should provide ample learning opportunities and exposure to the following career-relevant concepts...

  • Arduino Prototyping.
  • Object-Oriented Programming in C/C++ and JavaScript (REACT)
  • HTML/CSS, JavaScript, and other programming languages.
  • REST API and Microservice Architecture Patterns.
  • Database Design (potentially).
  • Network Driver Development and Wireless I/O.
  • Real-Time Data Collection and Data Mining Techniques.
  • Documentation.

Project Motivation

An IT-relevant hobby is a great way to learn new skills, develop a work portfolio, test new ideas, and connect with like-minded individuals. Modern IOT developers need a basic understanding of sensors, controls, drivers, compilers, UX/UI interfaces, databases, and network design to be effective in their roles. Single board computer design is a fun and affordable hobby that offers significant exposure to these practical IT concepts in the form of interesting projects and tutorials.

Project Goals and Assessments

This project has been broken down into several weekly sub-goals, designed for an estimated time expenditure of 3-4 hours per week. Project goals and proposed assessment criteria are presented in the list below.

Week 1: Research, Proposal Drafting, and Ordering Materials

  • Set up GitHub Pages website.
  • Develop the initial project idea from conception into a working plan.
  • Participate in a class discussion to refine the project ideas.
  • Identify helpful research materials (e.g., articles, YouTube videos, product reviews, blogs, forums, etc.)
  • Procure the project materials.

Week 2-5: Complete the Freenove Arduino Project Board Tutorials.

  • Learn how to develop Arduino hardware drivers and use the Arduino IDE by completing the FreeNove project board tutorials.
  • Post insights and weekly progress reports on GitHub and Piazza.
  • Document the learning experience and explain the useful concepts using photos, essays, project code, etc.

Week 6: Developing a Device Assembly Plan

  • Create a device blueprint using https://www.circuito.io or an equivalent software.
  • Integrate knowledge from research materials (YouTube videos, tutorials, papers, etc.) to develop a plan for assembling the device. Order any missing parts, if necessary (e.g., connectors, cables, mounts, chips, harnesses, etc.)
  • Develop a code and device testing plan for each stage of development.

Week 7: Assembling, calibrating, and Testing the Various Meter Functions.

  • Establish a USB programming connection between the Uno and/or NodeMCU a local computer.
  • Integrate the sensor probes (e.g., temp, pH, and TDS) and peripheral components to create a greenhouse testing device.
  • Monitor streaming sensor data collected accross a USB interface.

Week 8: Streaming Data with Bluetooth.

  • Install a Bluetooth module on the device.
  • Develop the firmware codebase to create a Bluetooth connection.
  • Download the Arduino IDE mobile phone application and connect to the device.
  • Observe streaming sensor data on a cell phone and document the result.
  • Perform regression testing on the rest of firmware codebase to make sure that older features still work.
  • Investigate the Arduino unit testing frameworks and begin developing a suite of regression tests.

Week 9: Streaming Data with Wi-Fi.

  • Install a Wi-Fi transceiver on the existing device.
  • Develop the firmware code to establish a Wi-Fi connection using a local area router and monitor the resource from a local computer.
  • Evaluate the device operation under the new firmware. Has the new configuration affected the performance of the device in any way? Was the Wi-Fi connection a success? Why or why not?

Week 10: Exporting Data in SQL, XML, and JSON

  • Explore data storage and back-up techniques, including SQLite, XML, and JSON storage formats. What section of the device interface appears to work best for extracting the data in real time? What equations should be used to extract the data into a more meaningful form? (e.g., “timed arithmetic average for pH sensor data.”)
  • Create a folder that will eventually contain a Python script for serving dynamic HTML, CSS, and JavaScript (i.e., a REST API).
  • Place the SQLite, XML, or JSON file(s) inside of the API folder. Program the firmware to overwrite a dependency file whenever the device collects a new measurement. Which data format (SQL, XML, or JSON) seems easiest to incorporate with the proposed API? Which data format appears least compatible with the proposed API? Explain.
  • Regression testing.

Week 11: Developing a Simple Web Page to Display Sensor Data.

  • Write a python script to serve dynamic single page web service using HTML/CSS/JavaScript combined with formatted data developed from week 10.
  • Render the API on a python simple server. Record a quick video and share the results on Piazza.
  • Ensure the script renders the page correctly and all back-up files and dependencies are maintained by the transmitting device.

Week 12: Documentation, Wrap-Up, and Presentation.

  • Review the project materials, compose final documents and/or videos, and apply final touches to the project documentation.
  • Complete the final presentation explaining how each section of the project went and share it with GitHub Pages and the rest of the class.

Risks to Completing the Project

This will be my first Arduino project, so I expect to encounter some technical difficulties along the way. Here are some of the risks to completing the project on time:

  • Broken or missing parts: Forgetting to purchase a power supply or special component, or needing to replace a broken piece at the wrong time could delay the project timeline by a week or so.
  • Running out of memory: The final device is intended to function as a standalone computer with local and wide area network connectivity. Single board computers contain less memory than a typical computer, however. Running out of on-board memory would require restructuring the code base and rolling development features, either of these outcomes would jeopardize the project timeline and outcome.
  • Choosing the right development tools for the job: It is difficult to know ahead of time every tool that will work best for this project.
  • Forgetting a project section: It is possible that I may have overlooked some critical detail in the project outline. There is also a chance that the project could be significantly more complicated than it is presented to be. Both risks need to be addressed in the project timeline.
  • Completing the project in the wrong order: Performing the development steps in the wrong order could lead to complex bugs involving electrical components. For example, a project task could involve parts or firmware that are incompatible with another section of the project. Troubleshooting a problem such as this could be time-consuming.

Mitigation Strategies

  • Broken or Missing Parts: You can find a list of purchased project materials here. After ordering the first round of parts, I estimated that the lead time is somewhere between 5 and 7 days for replacement parts. I plan to organize dry run roughly once a week, which should provide time to check the inventory before the next round of development. That way I can identify any missing or broken parts before they become an issue.
  • Running out of memory: Memory limitations are a legitimate concern when developing firmware on small computers, since there is often limited memory available in these devices. Recent advances in data storage technology could offer a method to help combat this problem however, and I plan to investigate USB storage expansion options if the occasion ever presents itself. Aside from expanding the physical memory of the device, I will also need to exercise common optimization techniques and search for opportunities to used more efficient algorithms to make better use of the space available.
  • Selecting the right development tools: I have done my best to outline the tools and resources I plan on using for each stage of the project outline. That said, there are likely to be some changes to this plan after I uncover details of building a transmitter device. I have provided some additional time in the first 5 weeks of the timeline to investigate the tools that I will need to complete the project (tools for developing an asychronous REST API, for example). Therefore, if a tool does not appear to be working as planned, the timeline should be flexible enough to be able to change tools and/or directions whenever this becomes necessary.
  • Completing project sections in the wrong order: Again, I have afforded myself a 5-week grace period that I can use to reassess the timeline and make changes as I identify them. The first clear objectives of this project seems to 1.) develop a website, and 2.) learn about the Arduino hardware and IDE. Finally, I will need to extract data generated by the device and display this information in a webpage, live-stream and/or statistically summarized format (I would like to investigate data warehousing and OLAP methodologies as well in this project). This is an R&D project, so I expect the connections between concepts to become much clearer as I investigate each project variable in turn.
